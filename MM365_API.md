## MM365 API Documentation

### Base URL and Authentication
- **Base URL**: `{base_url}` (generic; may change)
- **Auth options**:
  - Header: `Authorization: token <api_key>:<api_secret>`
  - Or Basic auth with user/password or session cookie
- **Content-Type**: `application/json`

### Standard Frappe REST Endpoints
- **List**: `GET {base_url}/api/resource/{Doctype}`
- **Retrieve**: `GET {base_url}/api/resource/{Doctype}/{name}`
- **Create**: `POST {base_url}/api/resource/{Doctype}`
- **Update (partial)**: `PUT {base_url}/api/resource/{Doctype}/{name}`
- **Delete**: `DELETE {base_url}/api/resource/{Doctype}/{name}`

Common query params:
- `fields=` JSON array of fields to return (e.g., `["name","first_name"]`)
- `filters=` JSON array of filter tuples (e.g., `[["MM365 Contacts","first_name","like","Joh%"]]`)
- `order_by`, `limit_start`, `limit_page_length`

### Discovering the Full Field List (Live Metadata)
To enumerate all fields, types, options, and flags directly from your site:
- `GET {base_url}/api/resource/DocType/MM365%20Contacts`
- `GET {base_url}/api/resource/DocType/MM365%20Realtors`

Or (alternative):
- `GET {base_url}/api/method/frappe.desk.form.load.getdoctype?doctype=MM365%20Contacts`
- `GET {base_url}/api/method/frappe.desk.form.load.getdoctype?doctype=MM365%20Realtors`

These endpoints return `fields` with `fieldname`, `label`, `fieldtype`, `reqd`, `read_only`, `options`, etc. Use them to keep this documentation in sync with your actual DocType definitions.

## MM365 Contacts
Doctype name: `MM365 Contacts` (URL-encoded: `MM365%20Contacts`)

### Fields
The following fields are known from the codebase and standard Frappe fields. Use the metadata endpoints above to retrieve any additional custom fields.

| Field | Type | Required | Read-only | Description | Allowed on Create | Allowed on Update |
| --- | --- | --- | --- | --- | --- | --- |
| `name` | Data | System | Yes | Auto-generated unique ID (DocType primary key) | No | No |
| `owner` | Data | System | Yes | Document creator | No | No |
| `creation` | Datetime | System | Yes | Created timestamp | No | No |
| `modified` | Datetime | System | Yes | Last modified timestamp | No | No |
| `modified_by` | Data | System | Yes | Last modified by | No | No |
| `docstatus` | Int | System | Yes | Document status (0/1/2) | No | No |
| `idx` | Int | System | Yes | Row index | No | No |
| `first_name` | Data | No | No | First name of contact; used in referral link generation (defaults to "contact" if missing) | Yes | Yes |
| `last_name` | Data | No | No | Last name of contact; used in referral link generation if present | Yes | Yes |
| `homeowners_referral_link` | Data | No | Business-managed | Auto-generated on insert/save/validate if blank | No (server sets) | No (server may overwrite/regenerate) |
| `homebuyers_referral_link` | Data | No | Business-managed | Auto-generated on insert/save/validate if blank | No (server sets) | No (server may overwrite/regenerate) |

> Note: Additional custom fields may exist in your instance. Fetch them via the DocType metadata endpoints and treat fields with `read_only: 1` as non-editable.

### Behavior and Business Rules
- On create and on every save/validate, if either referral link is missing, the server generates it. Treat both referral link fields as read-only.
- Referral links include a time-based unique identifier; manual override is discouraged.

### API Examples
- List
  - `GET {base_url}/api/resource/MM365%20Contacts?fields=["name","first_name","last_name","homeowners_referral_link","homebuyers_referral_link"]&limit_page_length=20`
- Retrieve
  - `GET {base_url}/api/resource/MM365%20Contacts/{name}?fields=["*"]`
- Create (referral links generated by server)
  ```bash
  curl -X POST "{base_url}/api/resource/MM365%20Contacts" \
    -H "Authorization: token <api_key>:<api_secret>" \
    -H "Content-Type: application/json" \
    -d '{"first_name":"Jane","last_name":"Doe"}'
  ```
- Update (partial)
  ```bash
  curl -X PUT "{base_url}/api/resource/MM365%20Contacts/{name}" \
    -H "Authorization: token <api_key>:<api_secret>" \
    -H "Content-Type: application/json" \
    -d '{"last_name":"Smith"}'
  ```
- Delete
  ```bash
  curl -X DELETE "{base_url}/api/resource/MM365%20Contacts/{name}" \
    -H "Authorization: token <api_key>:<api_secret>"
  ```

- Filter examples
  - By first name prefix: `GET {base_url}/api/resource/MM365%20Contacts?filters=[["MM365 Contacts","first_name","like","Jan%"]]&fields=["name","first_name"]`
  - With generated homeowners link: `GET {base_url}/api/resource/MM365%20Contacts?filters=[["MM365 Contacts","homeowners_referral_link","is","set"]]`

### Fields
The following fields are known from the codebase and standard Frappe fields. Use the metadata endpoints above to retrieve any additional custom fields.

| Field | Type | Required | Read-only | Description | Allowed on Create | Allowed on Update |
| --- | --- | --- | --- | --- | --- | --- |

### Behavior and Business Rules

- If `first_name` is missing, link generation is skipped; send `first_name` on create.

  ```bash
    -H "Authorization: token <api_key>:<api_secret>" \
    -H "Content-Type: application/json" \

  ```
- Update (partial)
  ```bash
  curl -X PUT "{base_url}/api/resource/MM365%20Realtors/{name}" \
    -H "Authorization: token <api_key>:<api_secret>" \
  ```

## Error Handling
- 200/201: Success
- 401/403: Authentication or permission issue
- 404: Not found
- 417/422: Validation failure
- 409: Duplicate name / unique constraint violation

## Notes
- Treat referral link fields as read-only. If you send them empty or omit them, the server ensures generation. If you try to overwrite them, the server logic may regenerate or ignore the change.
- Always prefer fetching the DocType metadata when building clients, to stay in sync with the actual field list and constraints.
